# Audit repository — 13 luglio 2026

## Esito

I due commit inviati dopo la PR del database JSON non hanno sovrascritto il nuovo motore. La versione corrente conserva `FileStore`, l'importazione dei programmi e l'inizializzazione dal database JSON.

La pulizia applicata in questo branch risolve invece i problemi di igiene e separazione dei dati emersi durante l'audit.

## Controlli eseguiti

- sintassi di tutti i file JavaScript con `node --check`;
- test del database e dell'importatore;
- test statico di privacy e confini Git;
- parsing dei file JSON e del manifest;
- assenza di file tracciati sotto `.vs/`, `.vscode/` e `node_modules/`;
- assenza di database personali tracciati sotto `data/user/` e `data/private/`;
- ricerca di marcatori personali e della vecchia configurazione AI nella parte distribuibile;
- verifica che il database predefinito sia dimostrativo;
- verifica delle regole principali di `.gitignore`.

## Decisioni

1. I dati personali sono protetti per directory e non per nome del file.
2. `data/user/` può contenere database con qualunque nome.
3. `data/organizer-data.json` è un esempio generico versionato.
4. Il nome convenzionale `data/user/organizer-data.json` abilita il caricamento automatico locale.
5. `package.json` e i lockfile non sono ignorati: se compariranno dovranno essere revisionati e versionati consapevolmente. Sono ignorati soltanto dipendenze, cache e log generati.
6. Il vecchio livello di persistenza applicativa in `localStorage` è stato rimosso dal runtime; rimangono soltanto preferenze non critiche del logger.
7. Il futuro progetto pubblico dovrà essere creato con una nuova cronologia Git.

## Esito per la futura versione general purpose

Lo stato corrente del branch è utilizzabile come base da importare nel nuovo repository pubblico perché codice, dati dimostrativi e dati utente hanno confini espliciti. Non si deve comunque rendere pubblico il repository attuale, la cui cronologia privata ha contenuto dati personali.
